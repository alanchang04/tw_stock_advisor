# AI 選股→買進→賣出 完整思考流程（2026-07-15 現況）

> 這份文件整理「AI 從一天開始到結束，怎麼想、怎麼做決定」的完整流程，取代 `TRADING_LOGIC.md`
> 舊版（2026-07-09，尚未包含多空辯論結構化契約、裁決問責 guardrail、決策軌跡、中型流動性股轉向）
> 作為「現在到底在跑什麼」的權威文件。舊版仍保留當歷史脈絡（回測方法論bug修正紀錄）。
>
> 全流程分兩層：**量化層**（候選池怎麼篩、怎麼評分、怎麼出場——全部可回測、寫死規則）
> 和 **推理層**（LLM 多空辯論+裁決——非決定性，负責解讀新聞/題材、給人話理由）。
> 出場完全不靠 LLM（LLM 掛了也要能賣），進場才需要 LLM。

## 0. 每天什麼時候跑、跑什麼

`run_pipeline.py --mode auto`（GitHub Actions 排程，每天台灣時間 21:00）依星期自動切換：

- **週日～週四**（隔天是交易日）：完整流程,含「產生隔日進場推薦」
- **週五、週六**：只做資料更新+出場檢查+週末策略回顧,不開新倉

完整流程（`mode_pipeline`）順序（每段都寫進 `execution_log`,決策軌跡頁可逐段看）：

```
0. 搶 pipeline 互斥鎖（P0-1）→ 避免排程與手動同時跑，寫壞同一份 DB
1. data_ingest    補齊缺漏交易日的股價/技術指標
2. quality_gate   資料品質規則檢查（單日斷點/法人落後/筆數異常）+ 來源信心分數
3. market_intel   新聞/YouTube/美股速覽/月營收/季度財報/ETF換股/聰明資金/每日彙整
4. sector_momentum 官方54類族群5日動能（僅供 LLM 參考，選股不設硬閘門）
5. fills          回帳「昨天掛的單」今天開盤有沒有成交
6. orders_exits   持倉出場檢查 → 掛「明日開盤賣出」委託（不依賴 LLM）
7. risk_gate      市場濾網（空頭時可選擇整天不開新倉）
8. factor_screen  候選股篩選+評分（量化層，見第1節）
9. debate_bull / debate_bear / judge   多空辯論+裁決（推理層，見第2、3節）
10. orders_entries 把裁決結果掛成「明日開盤買進」委託
10.5 practice_track 人類練習軌：純量化篩20檔，不進LLM（見第6節）
11. advisors       手動持倉建議 + 追蹤清單買點提醒（獨立於 AI 部位）
12. notify         組報告 → Telegram 推播（AI軌主報告 + 練習軌獨立訊息）
```

**成交時序**（很重要，決定了「訊號」跟「成交」不是同一價）：pipeline 收盤後
20:00~22:30 才跑，當天收盤價早就買不到了。所以是「今晚收盤算訊號 → 寫進
`pending_orders` 掛單 → 明晚 pipeline 用『明天的』開盤價 ±滑價 真正成交」。
這也是券商 API 的正確架構（委託 → 成交回報），`agent/broker.py` 把這層抽象出來，
預設 `PaperBroker`（紙上模擬）,`ShioajiBroker`（永豐,尚未接上線）走同一套介面。

## 1. 候選股篩選（量化層，`agent/stock_selector.py` + `agent/strategy.py`）

### 1.1 硬門檻（先把「不夠格」的股票濾掉）

2026-07-15 轉向「中型流動性股」後的門檻（`STRATEGY`，`agent/strategy.py`）：

| 門檻 | 值 | 目的 |
|---|---|---|
| 收盤價 | ≥15 元 | 排除雞蛋水餃股 |
| RSI(14) | 45~88 | 排除超弱/極端超買 |
| 成交量 | ≥500 張 | 排除完全沒人交易的股票 |
| **近5日均成交金額** | **≥2億/日** | **抗操控核心**：全市場約1948檔裡有1175檔<5千萬/日，
籌碼薄好拉抬，這道門檻直接濾掉——因為沒有市值資料，用「成交金額」代理，
比市值更貼近「抗操控」目標（低流動性才是主力好拉抬的根源） |

`use_hot_sector_gate=False`：候選池是**全市場**，不是「先選熱門族群再選股票」的
硬閘門（舊版本這樣做會把整年強勢的南亞科這種「族群不常進前5」的股票擋在門外）。
熱門族群仍會算出來，當作 LLM 的參考資訊（`get_hot_sectors`）。

### 1.1a' 流動性 OR 閘門（2026-07-16，中小型股×投信剛開始買）

2億/日門檻是抗操控核心，但也把「投信剛買進、還沒漲一大段的中小型股」擋在外面。
`agent/strategy.py apply_liquidity_gate()` 加一條替代路徑，只用於 AI 軌：
```
通過門檻 = 成交金額≥2億/日  OR  (投信新進場 且 成交金額≥3千萬/日)
```
「投信新進場」（`compute_new_entry_flag`）＝連買第1~2日、且當日買超量體≥50張——
跟獎勵「已經連買很多天」的 `w_invest_streak` 因子方向相反，這裡抓的是剛開始買、
還沒漲一段的訊號。投信有揭露義務、不是隨便的主力，機構真金白銀的新進場訊號可以
替代單純的成交金額規模，但仍留 3 千萬/日的最低下限，不是完全不設防。回測驗證
（同期間 A/B）：加了這條路徑後總報酬 +151%（原+124%）、勝率44.5%（原42.3%）、
最大回撤 -26.7%（原-30.6%）都改善，唯獨 Sharpe 略降 1.45（原1.53，年化波動變大，
納入部分較小型股的預期代價）。練習軌（PRACTICE_CFG）刻意不開這條路徑，維持單純。

### 1.1a 空方硬否決規則（2026-07-15，程式層級強制，不經 LLM 裁量）

在硬門檻通過、評分排序**之前**，再加一道「無條件」排除：`agent/strategy.py
compute_hard_vetoes()` 純函式檢查每檔候選：

- **乖離月線(MA20)超過±15%**：追高追太遠
- **帶量長上引線**：上影線≥振幅60% 且 當日量≥近20日均量2.5倍（沿用出場規則
  `exit_upper_wick` 同一組門檻）——疑似高檔出貨

觸發任一條就直接從候選池剔除，**LLM 連看都看不到**，不像一般 VETO 還能被裁決
引用數據駁回。這是刻意設計成「無條件」而不是寫在 System Prompt 裡靠 LLM 自己遵守——
跟空方JSON截斷讓guardrail完全失效是同一個教訓：不能只信任 LLM 自覺會照做規則，
真正的硬門檻要靠程式碼算出來、寫死執行。排除清單會記在 `candidates.attrs["hard_excluded"]`，
Streamlit 頁面與 Telegram 報告都會顯示「被排除了哪些、為什麼」，不會悄悄消失。

### 1.2 評分公式（`score_candidates`，決定候選池內的排序，取前 20 檔進辯論）

每個因子乘上權重相加，2026-07-15 決策「動能降權、買氣+成長升權」後的現值：

```
趨勢（中度，不追極端飆股）：
  相對強度 RS20（全市場20日報酬百分位）        × 1.5   ← 原3.0，降權
  多頭排列 MA5>20>60 連續天數（20日封頂）       × 1.5   ← 原2.0
  60日動能相對排名                             × 1.0   ← 原1.5，降權
買氣（可回測，題材的量化代理）：
  投信連續買超（需累計≥100張才算，5日封頂）    × 2.5   ← 原1.5，升權
  外資買超（布林值）                           × 1.5   ← 原1.0，升權
成長（可回測基本面）：
  月營收年增 >0                                × 2.0   ← 原1.0，升權
  月營收年增 >20%（加碼）                       × 1.0   ← 原0.5，升權
新增（2026-07-16，中小型股×投信剛開始買）：
  投信新進場（連買1~2日+當日量體達標）          × 2.0   （可回測）
  近期主動式ETF加碼/新增檔數（2檔封頂）         × 1.0   （輔助佐證，不進回測）
（技術面單日事件旗標：黃金交叉/突破/MACD正/RSI甜蜜帶 → 全部權重0，只用於出場）
```

**為什麼降動能、升買氣+成長**：舊「趨勢版」權重最高是相對強度(3.0)，會挑出「漲最兇
的極端動能股」——但這類股票近30日新聞覆蓋率常常是 0 則（媒體不報這種微型投機股），
辯論的多空雙方只能複述當日漲跌幅和 RSI，論證薄弱。轉向後把權重挪給「買氣」（投信/
外資買超，代表法人認同）跟「成長」（月營收年增，代表基本面撐得住），選出來的是
「有基本面撐的中度強勢股」而非「最兇的飆股」——這批股票新聞覆蓋率高得多（媒體
覆蓋在中價位段80檔最密集，不是只有大型股），辯論才真的有料可吵。

回測驗證（2025-06-10~2026-07-14 同一期間，2026-07-15 用更寫實的滑價30bp+成交量
天花板+跌停鎖死模擬重跑過）：新權重**全面優於**舊趨勢版——總報酬 +124%（舊+110%）、
Sharpe 1.53（舊1.38）、Calmar 3.73（舊3.32），回撤持平（-30.6% vs -30.5%）。
跟 0050 比 Sharpe/Calmar 仍有差距（0050幾乎不回撤），但主動選股本就該承擔比大盤高
的波動，重點是相對於「自己的前一版」全面進步。

### 1.3 餵給 LLM 的資料長什麼樣（`format_candidates_for_llm`）

每檔候選股會附上：收盤/漲跌/RSI/MACD、趨勢結構（RS20百分位/多頭排列天數/60日動能）、
基本面（月營收年增）、籌碼（三大法人/外資買賣超，單位已轉成「張」——DB 存的是股，
÷1000 才是張,這是踩過三次的地雷)、**投信新進場**標記、**近期主動式ETF加碼檔數**、
近30日相關新聞標題（`market_signals` 表比對 `related_stocks` 陣列）。**法人資料若
當日缺失/落後於股價日期，會明確標註「不可據此論證」**，不會讓 LLM 拿 0 當「無法人
支撐」的假論據（2026-07-13 曾因並發寫入讓候選法人全讀成0，空方據此寫出錯誤的強
論斷，這是修正後的保護）。

**多主動式ETF增碼**：`etf_watchlist` 已種6檔主動式ETF（統一2檔+群益/野村/復華/
國泰各1檔）。2026-07-17 起群益的兩檔（00982A/00992A）已改走**官網每日完整持股**
（`data_pipeline/fetchers/capitalfund_etf_fetcher.py`）——查證後確認主動式ETF依
法規每日揭露完整持股(PCF)，MoneyDJ月更只是它自己抓得慢，不是資料源頭限制。用
Playwright 攔截網路請求找到群益投信官網真實API：`POST .../CFWeb/api/etf/buyback`，
回傳當日完整持股（實測53/37檔，遠多於MoneyDJ前10大）。**這支API被Incapsula防
機器人WAF擋下，純requests會逾時吊住**，必須用真的headless瀏覽器過JS挑戰——這是
本專案唯一需要瀏覽器自動化的爬蟲，CI額外裝`playwright install chromium`。野村/
復華/國泰仍走MoneyDJ月更（資料已停留6週未更新），要提升時效性需要重複同樣的
devtools網路攔截偵查（每家網站格式不同），先驗證群益試點穩定再考慮擴展。ETF訊號
只當評分加分項，且**不進回測**（資料量太稀薄，跟新聞是同一類限制）。

## 2. 多空辯論（推理層，`agent/llm_advisor.py _run_debate`）

候選股評分完、篩出前20檔後，兩個角色各自獨立呼叫一次 LLM（Gemini 2.5 Flash），
**互不看對方的輸出**（單輪、非來回辯論——見第2.4節的方案A/B）：

### 2.1 多方研究員（Bull）
從候選中挑出最看好的 5 檔，每檔要求：
- `thesis`（買進論點，2-4句）：**必須引用至少兩類證據**——趨勢結構、基本面、
  籌碼、題材，禁止只複述當日漲跌幅和 RSI（那不是論點，是複讀資料）
- `preempt_rebuttal`：誠實寫出「反對這檔最強的理由是什麼、為何仍值得買」
- `evidence_fields`：列出實際引用的資料欄位名（方便日後做引用驗證，防幻覺）

輸出契約：`{"picks":[{"stock_id","stock_name","thesis","evidence_fields","preempt_rebuttal"}]}`

### 2.2 空方研究員（Bear）
對整批候選提出風險審查：
- `vetoes`：對個股的具體異議，分兩級——`veto`（基於數據有具體理由絕對不該碰，
  例：外資大額賣超疑似出貨、跌勢未止、資料異常）、`caution`（有風險但可控，
  例：RSI偏高但趨勢完整）。**明確禁止**把所有 RSI>70 一律打 veto——動能股本來
  就常在高檔，要看趨勢結構是否完好。
- `market_concerns`：族群集中度/整體市場疑慮（最多3條）

輸出契約：`{"vetoes":[{"stock_id","stock_name","severity","reason"}],"market_concerns":[...]}`

### 2.3 JSON 解析與截斷救援（真實踩過的坑）

2026-07-14 發現的真bug：空方輸出常被 `max_tokens` 切斷成不完整 JSON,
`json.loads()` 失敗 → 裁決收到 `bear_data=None` → 下面第3節的 guardrail 直接
提早跳出、完全不會攔截任何 VETO。這正好解釋了使用者觀察到的「多方幾乎都被裁決
照抄」。修法：
1. `max_tokens` 調大（bull 2600、bear 3200），prompt 收斂輸出長度
2. 新增 `_repair_truncated_json()`：補閉未結束的字串、補未閉合的括號；還是解析
   不出來就逐次砍掉最後一個逗號片段重試（丟掉不完整的最後一筆，保住前面完整的）
3. 若空方 JSON 真的完全救不回來，**顯性標註**「guardrail 本次未啟用」寫進決策
   軌跡，不會默默降級成看不出來的樣子

用真實故障 payload（3067字截斷）重播驗證：救回 19 條 vetoes（修復前是 0 條）。

### 2.4 為什麼是「單輪」不是「來回辯論」

參考業界的 TradingAgents 架構是「單股深入、多輪來回」（空方回應多方、多方再駁），
現行系統是「20檔一次掃、單輪、多空各寫一次不互看」。2026-07-15 決策：**先維持
現行單輪結構（方案A）**，理由：候選池已經因為流動性門檻縮小、新聞終於有料，
先觀察品質；且 **Gemini 免費版每日每專案硬限20次呼叫**，逐檔多輪辯論會直接爆量。
未來要升級的改法已經寫進 `_run_debate()` 的 docstring：拆成「單股辯論函式」+
外層迴圈跑候選池，每輪把對方上一輪發言餵回去要求正面回應，直到沒有新論點或
達輪數上限——但需要付費 key 或換一個沒有每日20次限制的模型才撐得住呼叫量。

## 3. 首席裁決 + 裁決問責 guardrail（`generate_recommendations` / `apply_judge_guardrail`）

### 3.1 裁決（Judge/CIO）
把候選資料 + 多空辯論結果一起餵給第三次 LLM 呼叫，扮演「首席投資長」,規則寫死在
`SYSTEM_PROMPT`：

1. 空方對某檔提出的每一條異議，都必須明確回應：**「接受」**（不選/降級）或
   **「駁回」**（必須引用候選資料中的具體數據說明為何異議不成立）
2. 空方標記 **VETO** 的股票，除非能以具體數據駁回，否則不得入選
3. 論證須綜合趨勢結構/基本面/籌碼/題材，不可只複述當日漲跌幅與RSI
4. 提供 `backups`（最多2檔遞補名單）——因為系統會用程式檢查剔除不合格的入選股

輸出：5檔 `recommendations`（含 `objections_addressed` 逐條回應空方異議）+
`backups` + `market_summary`。

### 3.2 程式化 guardrail（不信任 LLM 自覺，`apply_judge_guardrail`）

2026-07-13 實錘：裁決曾經 5/5 照抄多方推薦、把空方 VETO 直接貼成 `risk_note`
照樣入選——**光靠 prompt 要求 LLM「自己」遵守規則是不夠的**，所以用純函式在
程式層再檢查一次：

```
被空方 VETO 的入選股：
  → objections_addressed 裡有「駁回」+ 具體理由(≥10字)？
    有 → 保留
    沒有 → 剔除，從 backups 遞補（遞補股同樣要過這關，沒得補就誠實地少一檔）
```

這是「軟模式」——VETO 不是绝对不能選，而是「選了就必須付出具體反駁的代價」，
逼裁決真的去讀空方的論點而不是照抄多方。攔截紀錄（剔除/遞補/遞補被拒）會寫進
決策軌跡，Telegram 報告與「🔍 決策軌跡」頁都看得到。

### 3.3 重試與容錯
裁決呼叫最多重試4次（15s/30s/45s 退避），區分暫時性錯誤（503/429/RateLimit）
跟解析失敗（非決定性，重生一次常就好）。全部失敗回空結果,當天只有出場提醒
沒有新推薦（不會讓整條 pipeline 掛掉）。

## 4. 下單：掛單 → 隔日開盤成交（`agent/portfolio.py` + `agent/broker.py`）

裁決出的5檔不會直接建倉，而是走「委託→成交回報」兩階段（`PaperBroker`）：

1. **今晚**（收盤後 pipeline 跑）：`queue_entries()` 把推薦寫進 `pending_orders`
   （`side='buy'`，訊號價=今日收盤價）。風控守門員在這裡就先擋：持倉數+已掛未成交
   買單數 若已達 `max_open_positions`（預設10檔），直接不掛新單。
2. **明晚**（下一次 pipeline 跑）：`fill_pending_orders()` 用「明天的開盤價」
   成交——買價 = 開盤價 ×(1+滑價)，賣價 = 開盤價 ×(1-滑價)（滑價目前是假設值
   10bp，待實盤校準）。開盤價缺（停牌）→ 買單直接取消不追價、賣單順延繼續掛著。
   買單掛超過10天沒成交 → 作廢（不追過期訊號，避免連假後補一個過時訊號）。
   成交後才真正寫進 `positions` 表（`status='open'`）。

**資金/風險管理**（`suggest_shares`，1%風險法則）：單筆最大虧損 = 資金×1%，
用停損距離（8%）反推張數；同時有「集中度天花板」——單一部位市值不超過
資金÷5（`pick_top_n`）。

## 5. 持倉期間：出場 12 條規則（`agent/strategy.py decide_exit`，完全不靠 LLM）

**這是刻意設計**：出場不依賴 LLM，即使 Gemini 額度用完/API掛掉，賣出提醒還是
一定會發（`daily_runner.py` 把出場檢查放在 LLM 呼叫之前，且獨立於 LLM 是否成功）。
每天對每個持倉檢查以下規則，任何一條觸發就出場（依「先保護本金」順序回報）：

| # | 規則 | 說明 |
|---|---|---|
| 1 | 停損 | 收盤跌破進場價 -8% |
| 2 | 固定停利 | 預設**關閉**（消融測試用，見#3） |
| 3 | 分級移動停利 | 峰值獲利≥50%後才啟動，回落25%才出場——「寬backstop」，
平時不介入（交給#5判斷趨勢），只防噴出後急崩把大獲利全吐回 |
| 4 | KD高檔死叉+MACD轉負 | K≥80且死叉、MACD柱同步轉負 |
| 5 | **均線死亡交叉**（主要出場） | MA5跌破MA20＝短期趨勢轉弱才走——這是「趨勢騎乘」
策略的核心，讓飛天股整段大波段吃下來 |
| 6-7 | 跌破MA20/MA5 | 預設關閉（交給#5，避免太敏感剁碎飛天股） |
| 8 | 跌破前波低點 | 30日內找左右各5根都更低的樞紐 |
| 9 | 跌破前3根實體棒底部 | |
| 10 | 跌破最近大K棒底部 | 漲跌幅≥3%的K棒 |
| 11 | 長上引線爆量 | 上影線≥振幅60%且量≥均量2.5倍——出貨訊號 |
| 12 | 持有到期 | 預設**關閉**（`max_hold_days=0`）——趨勢還在就一直抱 |

**為什麼是死亡交叉主導、不是固定停利**：walk-forward 回測比較過多種出場設計，
死亡交叉騎乘（E4）讓飛天股單次整段抱到（實測3491+103%/6274+97%/3026+120%），
全期報酬反而贏過0050同期表現；跌破週線（E5）雖然回撤更小，但會把飛天股剁成
+2~15%的碎片、全期報酬明顯較差——「風險小但報酬也小到不如直接買0050」，
使用者親自在兩者間選了E4（願意承擔更大回撤換抱住大波段）。

**市場濾網**：大盤代理（預設0050）收盤 vs MA60，空頭時（`market_filter`啟用）
自動把「死亡交叉」出場規則強制打開，即使原本設定關閉——空頭時多一層保護。

出場一樣走掛單機制：`queue_exits()` 今晚掛「明日開盤賣出」，`fill_pending_orders()`
明晚用開盤價成交，扣手續費(14.25bp×58折)+證交稅(30bp)算出淨報酬存回
`positions.net_return_pct`。

**回測/實盤帳本的寫實度限制**（2026-07-15 升級，`agent/strategy.py` / `agent/backtest.py`）：
- **滑價**：單邊 30bp（原 10bp），回測與 `PaperBroker` 共用同一個 `SLIPPAGE` 常數，
  仍是假設值不是量測值，待階段3實盤成交回報校準。
- **成交量天花板**：單筆張數 ≤ 近5日均量 1%（`suggest_shares(avg_volume=...)`），避免
  買到市場承載不了的量。30萬小資金規模下通常不會真的卡到，是資金放大後的護欄。
- **跌停鎖死模擬**（回測限定，`is_limit_locked()`）：開盤跌幅≤-9.5%且當日量<近5日均量
  10% → 判定「排隊賣不掉」，賣單/買單順延到下一交易日，不強制成交（避免回測低估
  真實跌停鎖死時的回撤）。

## 6. 人類交易員練習軌（2026-07-15 新增，純量化、完全不進 LLM）

跟第1~5節的「AI 軌」平行、互不影響的**第二條軌道**：目的不是幫使用者選股，而是讓
使用者練「波段操作 40% 勝率也能小賠大賺」的量化交易心態——每天收盤後純量化篩出
20 檔，使用者自己拿去 TradingView 只看 20MA 和成交量（刻意隱藏新聞/籌碼），練習
判斷突破進場、月線出場，不是抄 AI 軌的推薦。

`agent/stock_selector.py get_practice_candidates()` 重用 `get_candidate_stocks()`
的篩選/因子計算機制，只換一份權重與門檻（`PRACTICE_CFG`，`agent/strategy.py`）：

| 硬門檻 | 值 |
|---|---|
| 收盤價 | ≥15元（同AI軌） |
| 近5日均成交金額 | ≥2億/日（同AI軌） |
| **股價必須站上月線(MA20)** | 練習軌特有 |

評分只看三個純量化因子（技術面/動能/題材全部關閉，權重0）：
```
投信連續買超（籌碼優勢） × 2.5
多頭排列天數（趨勢優勢） × 1.5
月營收年增 >20%（基本面優勢） × 2.0
```
輸出前20檔到 Streamlit「🎯 練習軌」頁（表格形式）+ **獨立** Telegram 訊息（跟AI軌
主報告分開推播，內容不互相干擾）。同樣會套用第1.1a節的空方硬否決規則排除。

## 9. 誠實的限制與已知缺口

- **新聞/題材無法回測**：`market_signals`新聞的`created_at`集中在2026-06-30後，
  沒有逐日歷史存檔；更本質的是「當時市場怎麼解讀這則新聞」是當下的sentiment，
  事後補不出來。新聞只能當即時LLM判斷的輸入，不能用來證明歷史績效。
- **財報只有當期最新一季**：`financials_fetcher.py`用的TWSE/TPEX OpenAPI端點
  只回傳當期快照，不支援查歷史季度（MOPS正式查詢頁有歷史但有bot防護會403擋
  直接爬蟲，已實測）。所以基本面因子目前只能用月營收做回測，完整財報（EPS/
  ROE/毛利率等）只能供即時辯論引用，無法回測。
- **guardrail是軟模式**：VETO不是絕對否決，是「選了就要具體駁回」——如果裁決
  真的能引用數據反駁，VETO股仍會入選。這是刻意設計（避免空方一句話就否決掉
  真正有理由的多方論點），但也代表guardrail無法100%排除主觀判斷失準。硬否決
  規則（第1.1a節）是唯一真正「無條件」的排除，其餘都留給裁決判斷空間。
- **辯論仍是單輪**：兩個角色互不看對方輸出，不是真正的來回攻防，見第2.4節。
- **回撤仍比0050大**：主動選股承擔比大盤高的波動是選股邏輯的必然代價，不是bug。
- **Gemini免費版額度**：每日每專案20次呼叫硬限，是任何要拉高辯論強度/召回次數
  的設計都要面對的真實約束。
- **回測寫實度仍是近似**：跌停鎖死的判定門檻（跌幅≤-9.5%且量<均量10%）是合理
  的代理指標，不是真實委託簿深度模擬；滑價30bp/成交量天花板1%都是假設值，
  待階段3實盤成交回報校準才是真數字。
- **練習軌不是「更準的選股」**：它是刻意簡化到只剩3個純量化因子、拿掉LLM/新聞，
  目的是給使用者一個「乾淨、可歸因」的練習場，不是宣稱這比AI軌準——兩軌沒有
  互相比較績效的設計，也不該被這樣使用。
